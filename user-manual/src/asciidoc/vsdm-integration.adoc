
== How to integrate VSDM services

=== Scenario 1: Replace the Resource Server behind ZetaGuard

If you only want to replace the Resource Server (keeping ZetaGuard as PEP/proxy), follow these steps.

==== Prerequisites

* The external resource server must be started and reachable from the Testhub instance.

==== Steps

* Make sure the vsdm server simulation is not started in the compose setup.

[source,yaml]
----
# File: compose-vsdm-services.yaml (or the compose file that contains the vsdm server)
# Example: comment out the block for the vsdm-server
# vsdm-server:
#   image: ...
#   ports:
#     - "80:8080"
----

* In the Testhub compose settings: under the block that describes the PEP (e.g. `vsdm-zeta-pep`), remove or comment out the dependency on the internal `vsdm-server` (depends_on).

[source,yaml]
----
# Example: in compose-vsdm-services.yaml or compose-local.yaml
# under vsdm-zeta-pep:
#   depends_on:
#     - vsdm-server
# => comment out or remove
----

* Configure the PEP so that it forwards to the external (real) Resource Server.

Edit the `vsdm-zeta-pep.conf` file and replace the `proxy_pass` line:

[source,conf]
----
# Before (local vsdm-server in compose):
proxy_pass http://vsdm-server:80/;

# After (real resource server):
proxy_pass http://URL_TO_REAL_SERVER/;
----

Tip: Pay attention to the trailing `/` and to TLS (https://...) when required.

=== Scenario 2: Replace the complete domain service (ZetaGuard + Resource Server)

If the complete domain service (ZetaGuard and the Resource Server) is operated externally, follow these steps.

==== Prerequisites

* The external domain service (ZetaGuard + Resource Server) must be started and reachable from the Testhub instance.

* The Zeta Guard must be configured to accept the PoPP tokens issued by the Testhub.
  The pep_popp_issuer must be configured to be either the same external PoPP server used by the Testhub or an instance of the
  Popp-Token-Generator that is reachable from the ZetaGuard. The ZetaGuard uses the `pep_popp_issuer` configuration to validate
  the signature of the incoming PoPP tokens.

==== Steps

* Adjust the configuration of the vsdm-client to point to the external domain service.

In `compose-local.yaml` define the variable `VSDM_RESOURCE_SERVER_URL` with the URL of the external domain service (ZetaGuard).

[source,yaml]
----
- VSDM_RESOURCE_SERVER_URL: <Url of the external domain service (ZetaGuard)>
----

* For testing use an eGK image that contains valid data (IKNR, KVNR) for the external domain service.

In `compose-popp-services.yaml` define the variable `VIRTUAL_CARD_IMAGE_FILE` with the URL of the valid eGK card image.

[source,yaml]
----
- VIRTUAL_CARD_IMAGE_FILE: /data/cards/<applicable-egk-card-image>.xml
----

Also, configure the VSDM2 test feature files to use the correct card image.

Optional: to make the tests start up faster, comment out all the components in the compose files that are not needed for the
VSDM2 tests: vsdm-server, popp-server and their ingresses as well as their associated pep and pdp components.

=== Notes / Troubleshooting

* After changes to compose files always restart the affected containers (e.g. `docker compose down && docker compose up -d` for the changed services).
* Test reachability of the external service from the Testhub instance (e.g. `curl` from the host/container).
* Pay attention to CORS, authentication and TLS settings if clients report errors when accessing the external service.
